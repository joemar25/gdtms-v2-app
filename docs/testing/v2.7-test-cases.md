## 1. FAILED_DELIVERY Status Mapping

### 1.1 `DeliveryStatus.fromString` ‚Äî accepts both values

```dart
// lib/core/models/delivery_status.dart

test('fromString maps FAILED_DELIVERY to failedDelivery', () {
  expect(DeliveryStatus.fromString('FAILED_DELIVERY'), DeliveryStatus.failedDelivery);
});

test('fromString maps RTS (legacy) to failedDelivery', () {
  expect(DeliveryStatus.fromString('RTS'), DeliveryStatus.failedDelivery);
});

test('fromString is case-insensitive for both aliases', () {
  expect(DeliveryStatus.fromString('failed_delivery'), DeliveryStatus.failedDelivery);
  expect(DeliveryStatus.fromString('rts'), DeliveryStatus.failedDelivery);
});

test('toApiString emits FAILED_DELIVERY (v2.7 contract)', () {
  expect(DeliveryStatus.failedDelivery.toApiString(), 'FAILED_DELIVERY');
});

test('toApiString never emits RTS after v2.7 upgrade', () {
  for (final status in DeliveryStatus.values) {
    expect(status.toApiString(), isNot('RTS'));
  }
});

test('displayName shows Failed Delivery for failedDelivery', () {
  expect(DeliveryStatus.failedDelivery.displayName, 'Failed Delivery');
});
```

## 2. DELIVERY_IMMUTABLE 409 Handling

### 2.1 Machine-readable code takes priority

```dart

    'success': false,
    'message': 'Item is immutable.',

  expect(op.lastError, startsWith('Resolved:'));

  // Previously only DELIVERED intent was auto-resolved ‚Äî v2.7 allows all statuses
  ...
  expect(op.status, 'synced');
});
```

## 3. Retry-After 429 Throttling

```dart
// SyncManager ‚Äî 429 handling

test('respects Retry-After header before retrying', () async {
  fakeApi.stubPatch(
    barcode: 'X1',
    response: ApiRateLimited('Too Many Attempts.', retryAfterSeconds: 2),
  );
  fakeTime.startTracking();

  await syncManager.processPendingQueue();

  // At least 2s must have elapsed before the retry was attempted
  expect(fakeTime.elapsedSeconds, greaterThanOrEqualTo(2));
});

test('falls back to 60s delay when Retry-After is absent', () async {
  fakeApi.stubPatch(
    barcode: 'X1',
    response: ApiRateLimited('Too Many Attempts.'), // no retryAfterSeconds
  );
  // Verify the delay used was 60s (via captured delay duration)
  ...
});

test('marks op as failed after exhausting all retries on 429', () async {
  fakeApi.alwaysReturn(ApiRateLimited('Too Many Attempts.', retryAfterSeconds: 1));
  await syncManager.processPendingQueue();
  final op = await syncOpsDao.getById(opId);
  expect(op.status, 'failed');
  expect(op.retryCount, greaterThan(0));
});
```

## 7. v2.8 Business Logic

### Status transition rules

| From              | To                | Result                              |
| ----------------- | ----------------- | ----------------------------------- |
| `PENDING`         | `DELIVERED`       | ‚úÖ 200 OK                           |
| `PENDING`         | `FAILED_DELIVERY` | ‚úÖ 200 OK ‚Äî new attempt             |
| `PENDING`         | `OSA`             | ‚úÖ 200 OK                           |
| `FAILED_DELIVERY` | `FAILED_DELIVERY` | ‚úÖ 200 OK ‚Äî each is a new attempt, increments `rts_count` |
| `FAILED_DELIVERY` | `DELIVERED`       | ‚úÖ 200 OK ‚Äî re-delivery succeeded   |
| `DELIVERED`       | `DELIVERED`       | ‚úÖ 200 `SAME_STATUS_TRANSITION`     |
| `DELIVERED`       | `FAILED_DELIVERY` | ‚ùå 409 `DELIVERY_IMMUTABLE`         |
| `DELIVERED`       | `OSA`             | ‚ùå 409 `DELIVERY_IMMUTABLE`         |
| `OSA`             | any               | ‚ùå 409 `DELIVERY_IMMUTABLE`         |

```dart
test('FAILED_DELIVERY to FAILED_DELIVERY succeeds ‚Äî new attempt', () async {
  // Second failed attempt on same barcode ‚Äî not an error
  fakeApi.stubPatch(barcode: 'X1', response: ApiSuccess({'data': {'delivery_status': 'FAILED_DELIVERY'}}));
  await syncManager.processPendingQueue();
  final op = await syncOpsDao.getById(opId);
  expect(op.status, 'synced');
});

test('DELIVERED to DELIVERED returns 200 SAME_STATUS_TRANSITION', () async {
  // Server returns 200 with code SAME_STATUS_TRANSITION ‚Äî treated as success
  fakeApi.stubPatch(barcode: 'X2', response: ApiSuccess({'data': {'delivery_status': 'DELIVERED'}, 'code': 'SAME_STATUS_TRANSITION'}));
  await syncManager.processPendingQueue();
  final op = await syncOpsDao.getById(opId);
  expect(op.status, 'synced');
});
```

## 8. Risk List ‚Äî v2.8 Backward Compatibility

| Risk | Impact | Mitigation |
| ---- | ------ | ---------- |
| Old app versions still send `RTS` | Server must continue accepting `RTS` during rollout | Backend v2.8 accepts both; no action needed on mobile |
| `delivery_status = 'RTS'` rows in SQLite before upgrade | Items counted/queried with old IN clauses | DB v11 migration converts all to `FAILED_DELIVERY` |
| `deleteOldSynced` previously used lowercase `IN ('rts', 'osa')` | Records were never purged ‚Äî storage leak | Fixed with `COLLATE NOCASE IN (...)` in v2.7; `'RTS'` removed in v2.8 |
| Sync queue `payloadJson` with `delivery_status=RTS` | sync_screen RTS attempt counter missed them | Counter now checks `FAILED_DELIVERY \|\| RTS` |
| Legacy payout API responses with `delivery_status=RTS` | `defaultRtsVerif` was always `'unvalidated'` | Both `FAILED_DELIVERY \|\| RTS` checked in payout screens |
| `409 auto-resolve` only covered `DELIVERED` intent before v2.7 | `FAILED_DELIVERY` intent on immutable item landed as `conflict` loop | `DELIVERY_IMMUTABLE` code now auto-resolves for any intent |
